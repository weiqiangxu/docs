## 一、分布式事务概念

### 什么是分布式事务？

**分布式事务**是指事务的参与者、支持事务的服务器、资源服务器以及事务管理器分别位于不同的分布式系统的不同节点之上。

```mermaid
graph TD
    A[客户端] --> B[事务管理器<br/>协调者]
    
    B --> C[服务A<br/>数据库A]
    B --> D[服务B<br/>数据库B]
    B --> E[服务C<br/>数据库C]
    
    C --> F[本地事务]
    D --> G[本地事务]
    E --> H[本地事务]
    
    style B fill:#e1f5fe
    style C fill:#c8e6c9
    style D fill:#c8e6c9
    style E fill:#c8e6c9
```

### 为什么需要分布式事务？

**单体应用时代：**
- 单一数据库，使用本地事务（ACID）
- 通过数据库的事务机制保证数据一致性

**微服务时代：**
- 业务拆分为多个服务
- 每个服务有自己的数据库
- 跨服务的业务操作需要保证数据一致性
- 本地事务无法解决跨数据库的一致性问题

## 二、理论基础

### CAP 定理

> **一致性（Consistency）、可用性（Availability）、分区容忍性（Partition Tolerance）三个特性最多只能同时满足其中两个**

```mermaid
graph TD
    A[CAP定理] --> B[CA: 一致性+可用性]
    A --> C[CP: 一致性+分区容忍]
    A --> D[AP: 可用性+分区容忍]
    
    B --> B1[传统单机数据库<br/>RDBMS]
    C --> C1[分布式数据库<br/>MongoDB/HBase]
    D --> D1[分布式缓存<br/>Cassandra/DynamoDB]
    
    style A fill:#e1f5fe
    style B fill:#c8e6c9
    style C fill:#fff3e0
    style D fill:#ffcdd2
```

**详细说明：**

| 组合 | 说明 | 典型系统 | 应用场景 |
|------|------|---------|---------|
| **CA** | 放弃分区容忍性，单机系统 | MySQL、PostgreSQL | 传统单机应用 |
| **CP** | 放弃可用性，保证一致性 | HBase、MongoDB | 金融系统、配置中心 |
| **AP** | 放弃一致性，保证可用性 | Cassandra、DynamoDB | 社交网络、推荐系统 |

**CAP 权衡：**

1. **多节点为了数据一致性，数据同步阻塞服务，可用性就会变差**
2. **多节点为了高可用，但数据同步不及时，一致性变差**
3. **多节点因为部分节点连接中断，而无法正确提供服务，此为容忍性**

### BASE 理论

> **Basically Available（基本可用）、Soft State（软状态）、Eventually Consistent（最终一致性）**

```mermaid
graph LR
    A[BASE理论] --> B[基本可用]
    A --> C[软状态]
    A --> D[最终一致性]
    
    B --> B1[系统出现故障时<br/>允许损失部分可用性]
    C --> C1[允许系统存在<br/>中间状态]
    D --> D1[经过一段时间后<br/>最终达到一致]
    
    style A fill:#e1f5fe
    style B fill:#c8e6c9
    style C fill:#fff3e0
    style D fill:#c8e6c9
```

**BASE vs ACID：**

| 特性 | ACID | BASE |
|------|------|------|
| **一致性** | 强一致性 | 最终一致性 |
| **可用性** | 较低 | 较高 |
| **性能** | 较低 | 较高 |
| **复杂度** | 简单 | 复杂 |
| **适用场景** | 传统数据库 | 分布式系统 |

### 1. 两阶段提交（2PC）

**核心思想：** 通过协调者统一调度所有参与者的行为

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者1
    participant P2 as 参与者2
    
    Note over C: 阶段一: Prepare
    C->>P1: Prepare请求
    C->>P2: Prepare请求
    
    P1->>P1: 执行事务<br/>不提交
    P2->>P2: 执行事务<br/>不提交
    
    P1->>C: Yes/No
    P2->>C: Yes/No
    
    Note over C: 阶段二: Commit/Rollback
    alt 所有参与者都Yes
        C->>P1: Commit请求
        C->>P2: Commit请求
        P1->>P1: 提交事务
        P2->>P2: 提交事务
    else 有参与者No
        C->>P1: Rollback请求
        C->>P2: Rollback请求
        P1->>P1: 回滚事务
        P2->>P2: 回滚事务
    end
```

**优点：**
- 强一致性保证
- 实现相对简单
- 业界成熟方案

**缺点：**
- 同步阻塞问题
- 单点故障问题
- 数据不一致风险（协调者故障时）

**适用场景：**
- 对一致性要求极高的场景
- 数据库层面的分布式事务

### 2. 三阶段提交（3PC）

**核心思想：** 在2PC基础上增加CanCommit阶段，引入超时机制

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者1
    participant P2 as 参与者2
    
    Note over C: 阶段一: CanCommit
    C->>P1: CanCommit请求
    C->>P2: CanCommit请求
    
    P1->>C: Yes/No
    P2->>C: Yes/No
    
    Note over C: 阶段二: PreCommit
    alt 所有参与者都Yes
        C->>P1: PreCommit请求
        C->>P2: PreCommit请求
        
        P1->>P1: 执行事务<br/>不提交
        P2->>P2: 执行事务<br/>不提交
        
        P1->>C: Ack
        P2->>C: Ack
        
        Note over C: 阶段三: DoCommit
        C->>P1: DoCommit请求
        C->>P2: DoCommit请求
        
        P1->>P1: 提交事务
        P2->>P2: 提交事务
    else 有参与者No
        C->>P1: Abort请求
        C->>P2: Abort请求
    end
```

**优点：**
- 减少阻塞时间
- 引入超时机制
- 参与者可自主决策

**缺点：**
- 可能出现数据不一致
- 实现复杂
- 网络开销大

**适用场景：**
- 对可用性要求较高的场景
- 需要减少阻塞时间的场景

### 3. 可靠事件队列

**核心思想：** 通过消息队列实现最终一致性

```mermaid
graph LR
    A[服务A] --> B[消息队列]
    B --> C[服务B]
    
    A --> A1[执行本地事务]
    A1 --> A2[发送消息到MQ]
    
    B --> B1[持久化消息]
    B1 --> B2[投递给消费者]
    
    C --> C1[接收消息]
    C1 --> C2[执行本地事务]
    C2 --> C3[发送ACK]
    
    style A fill:#e1f5fe
    style B fill:#fff3e0
    style C fill:#c8e6c9
```

**实现方式：**

```mermaid
sequenceDiagram
    participant A as 服务A
    participant MQ as 消息队列
    participant B as 服务B
    
    A->>A: 执行本地事务
    A->>MQ: 发送消息
    
    alt 发送成功
        MQ->>MQ: 持久化消息
        MQ->>B: 投递消息
        B->>B: 执行本地事务
        B->>MQ: ACK确认
    else 发送失败
        A->>A: 重试发送
    end
```

**优点：**
- 实现简单
- 解耦服务
- 异步执行，性能好
- 最终一致性保证

**缺点：**
- 只有最终一致性
- 消息重复消费问题
- 需要处理幂等性

**适用场景：**
- 对实时性要求不高的场景
- 允许最终一致性的业务
- 高并发场景

**最大努力交付（Best-Effort Delivery）：**

```
可靠事件队列只要第一步业务完成了，后续就没有失败回滚的概念，只许成功，不许失败
靠着持续重试来保证可靠性的解决方案
```

### 4. TCC 事务（Try-Confirm-Cancel）

**核心思想：** 将业务逻辑拆分为三个阶段

```mermaid
graph TD
    A[TCC事务] --> B[Try: 尝试执行]
    A --> C[Confirm: 确认执行]
    A --> D[Cancel: 取消执行]
    
    B --> B1[资源检查<br/>资源预留]
    C --> C1[真正执行业务<br/>提交预留资源]
    D --> D1[取消执行<br/>释放预留资源]
    
    style A fill:#e1f5fe
    style B fill:#fff3e0
    style C fill:#c8e6c9
    style D fill:#ffcdd2
```

**详细流程：**

```mermaid
sequenceDiagram
    participant U as 用户服务
    participant B as 账户服务
    participant I as 库存服务
    
    Note over U: Try阶段
    U->>B: 冻结金额
    U->>I: 冻结库存
    
    B->>B: 检查余额充足<br/>冻结100元
    I->>I: 检查库存充足<br/>冻结1件商品
    
    B->>U: 冻结成功
    I->>U: 冻结成功
    
    Note over U: Confirm阶段
    U->>B: 扣减金额
    U->>I: 扣减库存
    
    B->>B: 扣减冻结的100元
    I->>I: 扣减冻结的1件商品
    
    B->>U: 扣减成功
    I->>U: 扣减成功
```

**示例代码：**

```java
public interface AccountService {
    
    // Try: 冻结金额
    boolean freezeAmount(String userId, BigDecimal amount);
    
    // Confirm: 扣减金额
    boolean deductAmount(String userId, BigDecimal amount);
    
    // Cancel: 解冻金额
    boolean unfreezeAmount(String userId, BigDecimal amount);
}

public class AccountServiceImpl implements AccountService {
    
    @Override
    public boolean freezeAmount(String userId, BigDecimal amount) {
        // 检查余额是否充足
        Account account = accountDao.findByUserId(userId);
        if (account.getBalance().compareTo(amount) < 0) {
            return false;
        }
        
        // 冻结金额
        account.setFrozenAmount(account.getFrozenAmount().add(amount));
        account.setBalance(account.getBalance().subtract(amount));
        accountDao.update(account);
        
        return true;
    }
    
    @Override
    public boolean deductAmount(String userId, BigDecimal amount) {
        Account account = accountDao.findByUserId(userId);
        // 扣减冻结金额
        account.setFrozenAmount(account.getFrozenAmount().subtract(amount));
        accountDao.update(account);
        return true;
    }
    
    @Override
    public boolean unfreezeAmount(String userId, BigDecimal amount) {
        Account account = accountDao.findByUserId(userId);
        // 解冻金额
        account.setFrozenAmount(account.getFrozenAmount().subtract(amount));
        account.setBalance(account.getBalance().add(amount));
        accountDao.update(account);
        return true;
    }
}
```

**优点：**
- 性能较好（资源锁定时间短）
- 最终一致性保证
- 灵活性高

**缺点：**
- 业务侵入性强
- 开发成本高
- 需要实现三个接口
- 需要处理幂等性

**适用场景：**
- 对性能要求较高的场景
- 业务逻辑复杂的场景
- 需要精细化控制资源的场景

**关键点：**
- 用户花钱了冻结金额，用户买书了就冻结书籍状态，讲究资源锁定
- 业务侵入性很强

### 5. SAGA 事务

**核心思想：** 将长事务拆分为多个本地事务，每个本地事务都有对应的补偿事务

```mermaid
graph TD
    A[SAGA事务] --> B[正向操作T1]
    B --> C[正向操作T2]
    C --> D[正向操作T3]
    
    D --> E{执行结果}
    
    E -->|成功| F[事务完成]
    E -->|失败| G[补偿操作C2]
    G --> H[补偿操作C1]
    
    style A fill:#e1f5fe
    style B fill:#c8e6c9
    style C fill:#c8e6c9
    style D fill:#c8e6c9
    style G fill:#ffcdd2
    style H fill:#ffcdd2
```

**两种恢复模式：**

```mermaid
graph LR
    A[SAGA恢复模式] --> B[正向恢复<br/>Forward Recovery]
    A --> C[反向恢复<br/>Backward Recovery]
    
    B --> B1[T1→T2→T3失败<br/>→T3重试→T4→T5]
    C --> C1[T1→T2→T3失败<br/>→C2→C1]
    
    style A fill:#e1f5fe
    style B fill:#c8e6c9
    style C fill:#fff3e0
```

**详细流程：**

```mermaid
sequenceDiagram
    participant O as 订单服务
    participant P as 支付服务
    participant I as 库存服务
    participant D as 配送服务
    
    Note over O: 正向流程
    O->>O: T1: 创建订单
    O->>P: T2: 扣款
    P->>P: 执行扣款
    P->>O: 扣款成功
    
    O->>I: T3: 扣减库存
    I->>I: 执行扣减
    I->>O: 库存不足!
    
    Note over O: 反向补偿流程
    O->>P: C2: 退款
    P->>P: 执行退款
    P->>O: 退款成功
    
    O->>O: C1: 取消订单
    O->>O: 订单已取消
```

**设计约束：**

1. **子事务必须原子行为**
2. **Ti和Ci都具备幂等性**
3. **Ti与Ci满足交换律**
4. **Ci必须能成功提交**

**优点：**
- 适合长事务
- 最终一致性保证
- 性能较好

**缺点：**
- 没有隔离性
- 补偿逻辑复杂
- 可能出现脏读

**适用场景：**
- 长事务场景
- 跨多个服务的业务流程
- 对性能要求较高的场景

**两种恢复模式：**

1. **正向恢复（Forward Recovery）**
   - Ti事务提交失败，则一直对Ti进行重试，直至成功为止
   - 执行模式为：T1，T2，…，Ti（失败），Ti（重试）…，Ti+1，…，Tn

2. **反向恢复（Backward Recovery）**
   - Ti事务提交失败，则一直执行Ci对Ti进行补偿，直至成功为止
   - 执行模式：T1，T2，…，Ti（失败），Ci（补偿），…，C2，C1

## 六、最佳实践

### 1. 幂等性设计

所有分布式事务方案都需要考虑幂等性：

```java
public class IdempotentService {
    
    private Map<String, Boolean> processedRequests = new ConcurrentHashMap<>();
    
    public boolean process(String requestId, Runnable action) {
        // 检查是否已处理
        if (processedRequests.containsKey(requestId)) {
            return true;
        }
        
        // 执行业务逻辑
        action.run();
        
        // 标记为已处理
        processedRequests.put(requestId, true);
        return true;
    }
}
```

### 2. 超时设置

合理设置超时时间，避免长时间阻塞：

### 3. 监控与告警

建立完善的监控体系：

```mermaid
graph TD
    A[监控体系] --> B[事务成功率]
    A --> C[事务耗时]
```

## 七、总结

### 核心要点

1. **分布式事务是微服务架构的必然挑战**
2. **没有完美的解决方案，需要根据业务场景选择**
3. **强一致性方案性能低，最终一致性方案性能高**
4. **幂等性、超时、监控是所有方案的共同要求**
